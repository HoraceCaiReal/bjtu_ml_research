# 日常开发协作流程

> 本文件规范两人在项目期间的日常协作方式，包括 Git 使用、GitHub Issues 讨论、代码提交规范等。

---

## 一、每日开发流程（双方相同）

### 开工前（必须做）

```powershell
git pull origin main
```

拉取对方最新的代码，避免覆盖他人工作。**养成习惯：开电脑前先 pull。**

### 开发中

1. 在 `src/` 或 `notebooks/` 中编写代码
2. 在本地 conda 环境 `bjtu_ml` 中运行测试，确保不报错
3. 如需新增依赖，先微信沟通，由一人统一修改 `environment.yml` 并推送

### 收工前（必须做）

```powershell
git add .
git commit -m "前缀: 描述你做了什么"
git push origin main
```

**注意**：
- 如果 `git push` 提示 `rejected`，说明对方在你开发期间推送了新代码。执行 `git pull origin main`，解决冲突后再 push。
- Jupyter Notebook 提交前建议先 `Kernel -> Restart Kernel and Clear Outputs`，然后保存。nbstripout 会自动清理残余输出，但手动清理更保险。

## 二、Git 使用规范

### 分支策略

本项目只有 `main` 一个分支，双方都直接 push 到 `main`。

**原因**：两人团队不需要多分支，简单即是效率。

### 提交信息规范

提交信息（commit message）必须有意义，方便日后查看历史。

| 前缀 | 用途 | 示例 |
|------|------|------|
| `feat:` | 新增功能/模块 | `feat: 添加 K-Means 聚类实验` |
| `fix:` | 修复 Bug | `fix: 修复数据划分时标签错乱` |
| `docs:` | 文档/报告更新 | `docs: 更新分工说明` |
| `refactor:` | 代码重构 | `refactor: 提取图像预处理函数` |
| `env:` | 环境/依赖变更 | `env: 添加 seaborn 依赖` |
| `test:` | 测试相关 | `test: 添加数据加载单元测试` |

**禁止的提交信息**：
- `update`（太模糊）
- `111` / `aaa`（无意义）
- `修改`（没有说明改了什么）

### 冲突处理

当 `git push` 失败，提示 `! [rejected] main -> main (fetch first)` 时：

```powershell
# 1. 拉取对方代码
git pull origin main

# 2. Git 会提示冲突文件，例如：
#   Auto-merging src/data_utils.py
#   CONFLICT (content): Merge conflict in src/data_utils.py

# 3. 打开冲突文件，搜索 "<<<<<<< HEAD" 标记
#    - HEAD 下方是你自己的代码
#    - ======= 下方是对方的代码
#    - >>>>>>> 下方是提交哈希
#    手动保留正确版本，删除所有标记行

# 4. 标记冲突已解决
git add .
git commit -m "merge: 解决 data_utils.py 冲突"
git push origin main
```

**减少冲突的技巧**：
- 开工前 `git pull`
- 不要同时修改同一个文件
- 通用函数抽到 `src/` 的独立 `.py` 文件中，减少 Notebook 直接冲突

## 三、GitHub Issues 讨论规范

### 为什么用 Issues？

- 所有讨论永久保存在仓库中，**老师可以直接查看**（评分项要求"详细的讨论记录"）
- 支持按标签分类、搜索、引用
- 可以在提交信息中引用（如 `fix #3` 自动关联 Issue #3）

### Issue 使用场景

| 场景 | 是否发 Issue | 示例标题 |
|------|-------------|----------|
| 技术方案讨论 | ✅ 必须发 | `[模型] CNN 输入尺寸选 128 还是 256` |
| 发现代码 Bug | ✅ 必须发 | `[Bug] data_utils.py 读取空文件夹时报错` |
| 依赖环境变更 | ✅ 必须发 | `[环境] 是否需要添加 albumentations 库` |
| 阶段性总结 | ✅ 建议发 | `[报告] 02_传统机器学习实验结果汇总` |
| 今天吃啥 | ❌ 不要发 | （发微信） |
| 帮忙带饭 | ❌ 不要发 | （发微信） |

### 创建 Issue 的步骤

1. 打开 https://github.com/lijiawei255/bjtu_ml_research/issues
2. 点击 `New issue`
3. 右侧 `Labels` 选择对应标签：
   - `讨论` — 一般性技术讨论
   - `数据` — 数据集、预处理相关问题
   - `模型` — 决策树、SVM、CNN、K-Means
   - `环境` — Anaconda、依赖安装问题
   - `报告` — 最终报告、分工说明、视频
   - `Bug` — 代码错误
4. 标题格式：`[标签] 简短描述`
5. 正文写清楚背景、问题、已尝试的方案
6. 讨论得出结论后，点击 `Close issue` 关闭

### 在提交信息中引用 Issue

```powershell
# 修复了 Issue #3 中提到的 Bug
git commit -m "fix: 修复 data_utils.py 空文件夹报错 (fix #3)"
```

GitHub 会自动在 Issue #3 的页面显示这条提交记录。

## 四、文件分工建议

为避免冲突，建议按以下方式分工：

| 负责人 | 主要文件 | 说明 |
|--------|----------|------|
| 协作者 | `src/data_utils.py` | 数据读取、划分、预处理 |
| 协作者 | `notebooks/01_数据探索与预处理.ipynb` | 数据探索可视化 |
| 协作者 | `notebooks/02_传统机器学习.ipynb` | 决策树、SVM、K-Means 实验 |
| 管理员 | `src/models/cnn.py` | CNN 模型定义 |
| 管理员 | `notebooks/03_深度学习.ipynb` | CNN 训练与调参 |
| **联合** | `notebooks/04_综合展示系统.ipynb` | 最终整合，双方共同维护 |

**src/ 下的通用模块**（如 `plot_config.py`、`evaluation/metrics.py`）：谁需要谁改，改前微信说一声。

## 五、代码质量 checklist

每次提交前自查：

- [ ] 代码能在本地 conda 环境 `bjtu_ml` 中正常运行
- [ ] 没有上传数据文件（`.jpg`、`.zip` 等）
- [ ] 没有上传模型权重（`.pth`、`.pkl` 等）
- [ ] Jupyter Notebook 已清除输出（或确认 nbstripout 会处理）
- [ ] 提交信息符合规范（`feat:` / `fix:` / `docs:` 等前缀）
- [ ] 如需新增依赖，已同步更新 `environment.yml`

## 六、常见问题速查

**Q：我 push 被拒绝了怎么办？**
```powershell
git pull origin main
# 解决冲突（如果有）
git add .
git commit -m "merge: 同步并解决冲突"
git push origin main
```

**Q：我不小心提交了数据文件到 Git 怎么办？**
```powershell
git rm --cached 文件名
git commit -m "fix: 移除误提交的数据文件"
git push origin main
```

**Q：我想看对方改了什么？**
```powershell
git log --oneline -10          # 最近 10 条提交
git show 提交哈希              # 查看某次提交的具体改动
git diff HEAD~1                # 查看最近一次提交的改动
```

**Q：Issues 里讨论完了，代码也改了，还需要做什么？**
- 关闭 Issue（点击 `Close issue`）
- 如果代码改动较大，在 Issue 里贴一句 "已在 commit 0e4c3fd 中修复"
