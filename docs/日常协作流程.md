# 日常开发协作流程

> 本文件规范团队在项目期间的日常协作方式，包括 Git 使用、GitHub Issues 讨论、代码提交规范等。

---

## 一、每日开发流程

### 开工前（必须做）

```powershell
# 1. 切换到 main 并拉取最新代码
git checkout main
git pull origin main

# 2. 从最新的 main 创建功能分支（分支名规范：feature/功能描述）
git checkout -b feature/你的功能名
```

> **为什么必须先 pull？** 确保你基于团队最新代码开始开发，避免后续合并时出现本可避免的冲突。

### 开发中

1. 在 `src/` 或 `notebooks/` 中编写代码
2. 在本地 conda 环境 `bjtu_ml` 中运行测试，确保不报错
3. **Jupyter Notebook**：提交前建议手动 `Kernel -> Restart Kernel and Clear Outputs`，nbstripout 会自动清理残余输出
4. 如需新增依赖，先微信沟通，由一人统一修改 `environment.yml` 并推送

### 收工前

#### 情况A：当天功能已完成 ✅

```powershell
# 1. 提交并推送功能分支
git add .
git commit -m "feat: 描述你做了什么"
git push origin feature/你的分支
```

然后到 **GitHub 网页** 操作：

| 步骤 | 操作 |
|------|------|
| **创建 PR** | 打开仓库主页 → 点击提示的 "Compare & pull request"，或进入 Pull requests 标签页 → New pull request。**base 选 `main`，compare 选你的 feature 分支**，填写标题和描述后提交 |
| **合并 PR** | 在 PR 页面点击 "Merge pull request" → "Confirm merge" |
| **删除远程分支** | 合并完成后，PR 页面会出现 "Delete branch" 按钮，点击删除远程 feature 分支 |

回到本地，同步并清理：

```powershell
# 3. 切回 main，拉取合并后的最新代码
git checkout main
git pull origin main

# 4. 删除本地 feature 分支（已完成使命）
git branch -d feature/你的分支
```

#### 情况B：当天功能未完成 🚧

```powershell
# 1. 提交当前进度作为备份，推送到远程
git add .
git commit -m "feat: 阶段性提交 — 简要描述当前进度"
git push origin feature/你的分支
```

> 此时**不要创建 PR**，也不要合并。功能分支会安全地保存在远程。

**下次继续开发时：**

```powershell
# 2. 切回你的 feature 分支
git checkout feature/你的分支

# 3.（建议）将 main 的最新改动合并到当前分支，减少最终合并时的冲突
git pull origin main
# 如有冲突，按下方「冲突处理」流程解决

# 4. 继续开发，全部完成后按「情况A」提交流程
```

> **说明**：Git 本身已经保存了你的所有 commit，不需要额外"暂存"操作。未完成的分支就是最好的暂存方式。

### 关键原则

| 原则 | 说明 |
|------|------|
| **先拉取再开发** | 每天开工前务必 `git pull origin main`，确保基于最新代码 |
| **PR 而非直接合并** | 通过 GitHub Pull Request 合并到 main，保留审查记录，方便追溯 |
| **合并后立即删分支** | 远程分支在 PR 页面一键删除，本地分支用 `git branch -d` 删除 |
| **未完成≠丢失** | 功能分支未完成时正常 commit + push 即可，下次接着在同一分支开发 |
| **恢复开发前同步 main** | 继续开发未完成分支时，先 `git pull origin main` 减少最终合并冲突 |
| **约定文件分工** | 团队成员提前协商好各自负责的文件模块，不同时修改同一文件，从源头避免冲突 |

## 二、Git 使用规范

### 分支策略

| 分支 | 用途 | 谁可以 push |
|------|------|------------|
| `main` | 稳定版本，始终可运行 | 各成员通过 merge 更新，**不直接 push** |
| `feature/xxx` | 功能开发 | 各自创建，各自 push |
| `fix/xxx` | Bug 修复 | 各自创建，各自 push |

**流程**：`main` → 切分支 → 开发 → push 分支 → GitHub 创建 PR → 网页合并 PR → 删除远程分支 → 本地拉取 main 并删分支。

### 提交信息规范

提交信息（commit message）必须有意义，方便日后查看历史。

| 前缀 | 用途 | 示例 |
|------|------|------|
| `feat:` | 新增功能/模块 | `feat: 添加 K-Means 聚类实验` |
| `fix:` | 修复 Bug | `fix: 修复数据划分时标签错乱` |
| `docs:` | 文档/报告更新 | `docs: 更新分工说明` |
| `refactor:` | 代码重构 | `refactor: 提取图像预处理函数` |
| `env:` | 环境/依赖变更 | `env: 添加 seaborn 依赖` |
| `test:` | 测试相关 | `test: 添加数据加载单元测试` |

**禁止的提交信息**：
- `update`（太模糊）
- `111` / `aaa`（无意义）
- `修改`（没有说明改了什么）

### 冲突处理

冲突有两种情况：

#### 情况 1：PR 页面提示有合并冲突（Can't automatically merge）

这说明你的 feature 分支与 main 最新代码有冲突，需要在本地解决：

```powershell
# 1. 确保在 feature 分支上
git checkout feature/你的分支

# 2. 拉取 main 最新代码并合并到当前分支
git pull origin main

# 3. Git 会提示冲突文件，例如：
#   Auto-merging src/data_utils.py
#   CONFLICT (content): Merge conflict in src/data_utils.py

# 4. 打开冲突文件，搜索 "<<<<<<< HEAD" 标记
#    - HEAD 下方是你自己的代码
#    - ======= 下方是 main 分支的代码
#    - >>>>>>> 下方是 main 分支的提交哈希
#    手动保留正确版本，删除所有标记行

# 5. 标记冲突已解决，推送更新
git add .
git commit -m "merge: 解决 data_utils.py 冲突"
git push origin feature/你的分支
```

> 推送后，GitHub 上的 PR 会自动更新，冲突标记消失，即可正常合并。

#### 情况 2：推送 feature 分支时被拒绝 `! [rejected]`

极少数情况（通常是因为你在多台电脑上开发同一分支），执行：

```powershell
git pull origin feature/你的分支
# 解决冲突（如果有）
git push origin feature/你的分支
```

**减少冲突的技巧**：
- 每天开工前 `git pull origin main`，让 feature 分支基于最新 main
- 继续开发未完成分支前，先 `git pull origin main` 同步
- **提前约定文件分工**：团队成员协商好各自负责的文件模块，不同时修改同一个文件
- 通用函数抽到 `src/` 的独立 `.py` 文件中，减少 Notebook 直接冲突

## 三、GitHub Issues 讨论规范

### 为什么用 Issues？

- 所有讨论永久保存在仓库中，**老师可以直接查看**（评分项要求"详细的讨论记录"）
- 支持按标签分类、搜索、引用
- 可以在提交信息中引用（如 `fix #3` 自动关联 Issue #3）

### Issue 使用场景

| 场景 | 是否发 Issue | 示例标题 |
|------|-------------|----------|
| 技术方案讨论 | ✅ 必须发 | `[模型] CNN 输入尺寸选 128 还是 256` |
| 发现代码 Bug | ✅ 必须发 | `[Bug] data_utils.py 读取空文件夹时报错` |
| 依赖环境变更 | ✅ 必须发 | `[环境] 是否需要添加 albumentations 库` |
| 阶段性总结 | ✅ 建议发 | `[报告] 02_传统机器学习实验结果汇总` |
| 今天吃啥 | ❌ 不要发 | （发微信） |
| 帮忙带饭 | ❌ 不要发 | （发微信） |

### 创建 Issue 的步骤

1. 打开 https://github.com/HoraceCaiReal/bjtu_ml_research/issues
2. 点击 `New issue`
3. 右侧 `Labels` 选择对应标签：
   - `讨论` — 一般性技术讨论
   - `数据` — 数据集、预处理相关问题
   - `模型` — 决策树、SVM、CNN、K-Means
   - `环境` — Anaconda、依赖安装问题
   - `报告` — 最终报告、分工说明、视频
   - `Bug` — 代码错误
4. 标题格式：`[标签] 简短描述`
5. 正文写清楚背景、问题、已尝试的方案
6. 讨论得出结论后，点击 `Close issue` 关闭

### 在提交信息中引用 Issue

```powershell
# 修复了 Issue #3 中提到的 Bug
git commit -m "fix: 修复 data_utils.py 空文件夹报错 (fix #3)"
```

GitHub 会自动在 Issue #3 的页面显示这条提交记录。

## 六、常见问题速查

**Q：我 push feature 分支被拒绝了怎么办？**
```powershell
git pull origin feature/你的分支
# 解决冲突（如果有）
git add .
git commit -m "merge: 同步并解决冲突"
git push origin feature/你的分支
```

**Q：我不小心提交了数据文件到 Git 怎么办？**
```powershell
git rm --cached 文件名
git commit -m "fix: 移除误提交的数据文件"
git push origin main
```

**Q：我想看对方改了什么？**
```powershell
git log --oneline -10          # 最近 10 条提交
git show 提交哈希              # 查看某次提交的具体改动
git diff HEAD~1                # 查看最近一次提交的改动
```

**Q：Issues 里讨论完了，代码也改了，还需要做什么？**
- 关闭 Issue（点击 `Close issue`）
- 如果代码改动较大，在 Issue 里贴一句 "已在 commit 0e4c3fd 中修复"
