## 二、初始化步骤（按顺序执行）

### Step 1：接受仓库邀请

管理员已在 GitHub 上邀请你为 Collaborator。请检查你的邮箱或 GitHub 通知，点击接受邀请。接受后你拥有直接向主仓库推送代码的权限。

### Step 2：克隆仓库

```powershell
# 打开 PowerShell 或 CMD，执行：
git clone https://github.com/lijiawei255/bjtu_ml_research.git
cd bjtu_ml_research
```

### Step 3：创建 Conda 环境

```powershell
# 根据 environment.yml 创建名为 bjtu_ml 的环境
conda env create -f environment.yml
```

> 如果报错 `UnavailableInvalidChannel`，说明你的 conda 配置了失效的镜像源。请删除 `%USERPROFILE%\.condarc` 文件后重试。

> **如果你的电脑没有 NVIDIA GPU**，创建环境后需要将 PyTorch 替换为 CPU 版本：
> ```powershell
> conda activate bjtu_ml
> pip uninstall torch torchvision -y
> pip install torch==2.1.1+cpu torchvision==0.16.1+cpu -f https://mirrors.aliyun.com/pytorch-wheels/cpu
> ```

这个过程需要 5-15 分钟，取决于网速和硬盘速度。看到 `Executing transaction: done` 和 `conda activate bjtu_ml` 提示才算完成。

### Step 4：激活环境并注册 Jupyter Kernel

```powershell
conda activate bjtu_ml

# 让 VSCode 能直接运行 Notebook（注册 Jupyter Kernel）
python -m ipykernel install --user --name=bjtu_ml --display-name "Python (bjtu_ml)"
```

### Step 5：配置 nbstripout

```powershell
nbstripout --install
```

这是**一次性操作**。此后每次你提交 `.ipynb` 文件时，Git 会自动清除 Notebook 中的 Cell 输出（避免仓库膨胀和冲突）。

### Step 6：配置数据集路径

```powershell
# 复制环境变量模板
copy .env.example .env
```

用 VSCode 打开 `.env` 文件，把 `CRACK_DATA_ROOT` 改成你电脑上数据集的实际路径。例如：

```bash
CRACK_DATA_ROOT=D:/课程数据/裂纹图像
```

> `.env` 文件已被 Git 忽略，不会上传仓库，各自独立配置。

### Step 7：VSCode 选择解释器

1. 用 VSCode 打开 `bjtu_ml_research` 文件夹
2. 按 `Ctrl+Shift+P`
3. 输入并选择 `Python: Select Interpreter`
4. 选择 `bjtu_ml (conda)`

### Step 8：验证环境

在 VSCode 中新建一个终端（确保终端左下角显示 `(bjtu_ml)`），运行：

```python
python -c "import torch, sklearn, cv2, ipywidgets; print(torch.__version__)"
```

应输出类似 `2.1.1+cu118` 的版本号（GPU 版）或 `2.1.1+cpu`（CPU 版），无报错。

如需验证 GPU 是否可用：
```python
python -c "import torch; print('CUDA available:', torch.cuda.is_available())"
```
有 GPU 的电脑应输出 `CUDA available: True`。

### Step 9：在 VSCode 中开始编辑

直接用 VSCode 打开 `.ipynb` 文件即可开始编辑。VSCode 内置 Jupyter 支持，无需额外启动服务。

## 三、整体协作流程概述

### 我们用什么协作？

| 工具 | 用途 |
|------|------|
| **Git + GitHub** | 代码版本管理，功能分支开发后合并到 main |
| **GitHub Issues** | 技术讨论、方案设计、问题记录 |
| **微信/QQ** | 即时沟通，开工前说一声避免冲突 |
| **Anaconda** | 统一 Python 环境，确保各成员依赖完全一致 |

### 协作模式

本团队只有团队成员，采用**功能分支 + main 合并**模式：

- 协作者直接克隆主仓库，**不需要 Fork**
- 各成员都有写权限，各自创建 `feature/xxx` 分支开发
- 完成后合并回 `main`，可本地 merge 或 GitHub 网页一键合并
- 没有 Pull Request 审核环节，但合并前可以互相看一下对方分支的代码
- 冲突靠**分支隔离** + **提前沟通**来避免

### 日常开发流程

```powershell
# 1. 同步 main
git checkout main
git pull origin main

# 2. 创建功能分支
git checkout -b feature/你的功能名

# 3. 开发、测试...

# 4. 提交并推送分支
git add .
git commit -m "feat: 描述"
git push origin feature/你的功能名

# 5. 合并到 main
git checkout main
git pull origin main
git merge feature/你的功能名
# 如有冲突，解决后 git add . && git commit -m "merge: 解决冲突"
git push origin main

# 6. 删除已合并分支
git branch -d feature/你的功能名
```

详细的提交规范、冲突处理、文件分工见 `docs/日常协作流程.md`。

## 四、环境如何为项目目标服务

本项目预计完成裂纹图像识别系统，涉及以下技术：

| 技术方向 | 所需环境支持 | 本环境配置 |
|----------|-------------|-----------|
| **数据处理** | OpenCV、PIL、NumPy、Pandas | `opencv-python==4.9.0.80`, `pillow=10.2`, `numpy=1.26`, `pandas=2.1` |
| **决策树 / SVM** | scikit-learn | `scikit-learn=1.4` |
| **K-Means 聚类** | scikit-learn | `scikit-learn=1.4` |
| **卷积神经网络 (CNN)** | PyTorch CUDA 11.8（自动回退 CPU） | `torch==2.1.1+cu118` + `torchvision==0.16.1+cu118` |
| **综合展示系统** | Jupyter + ipywidgets | `jupyterlab=4.0`, `ipywidgets=8.1` |
| **中文图表显示** | Matplotlib + 微软雅黑 | `matplotlib=3.8` + `src/plot_config.py` |
| **代码格式化** | Ruff | `ruff==0.15.11` + VSCode Ruff 插件 |
| **报告撰写** | Jupyter Notebook | `notebook=7.0` |

### 关键设计决策

1. **PyTorch 用 CUDA 版**：环境默认安装 `torch==2.1.1+cu118`，有 NVIDIA GPU 的电脑自动使用 GPU 加速训练；无 GPU 时 PyTorch 自动回退到 CPU，无需修改代码。如需纯 CPU 版可手动替换（见 Step 3 说明）。

2. **conda + pip 混合安装**：NumPy、Pandas、scikit-learn 等科学计算包用 conda 安装（自动处理 Windows DLL 依赖，更稳定）；PyTorch CUDA 版本和 OpenCV 用 pip 安装（conda 的 PyTorch 不支持指定 CUDA 版本，OpenCV 的 conda/pip 混装会引发 DLL 冲突）。

3. **nbstripout 自动清理**：Jupyter Notebook 的输出（图片、打印日志）体积巨大且容易产生 Git 冲突。nbstripout 在后台自动清除输出，各成员提交的 `.ipynb` 只保留代码和 Markdown，干净清爽。

4. **.env 管理数据路径**：各成员电脑路径不同，通过 `.env` 各自配置，代码中统一使用 `src.config.DATA_ROOT`，无需修改源代码。

5. **Ruff 统一代码风格**：一个工具同时提供格式化（替代 Black）和代码检查（替代 Flake8），各成员保存文件时自动格式化，避免 "你的代码缩进是 Tab 我的是空格" 的冲突。
