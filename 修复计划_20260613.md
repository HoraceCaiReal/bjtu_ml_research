## BUG-001: CNN 预训练权重架构不匹配 [严重]

### 根因分析

`.pth` 文件中的键名使用**旧架构命名** (`c1/c2/c3/c4/cls`)，而当前 `CrackCNN` 类使用**新命名** (`block1/block2/block3/block4/classifier`)。

**已验证**：50 个键一一对应，shape 完全匹配，仅名称不同。

```
旧(.pth): c1.0.weight, c1.1.weight, ..., c4.4.weight, cls.weight, cls.bias
新(代码): block1.0.weight, block1.1.weight, ..., block4.4.weight, classifier.weight, classifier.bias
```

### 修复方案: 加载时自动键名映射

**文件**: `src/gradio_app.py` 第 821-822 行

**当前代码**:
```python
if optimization == "pretrained" and cnn_path and cnn_path.exists():
    model.load_state_dict(torch.load(cnn_path, map_location=DEVICE))
```

**修改为**:
```python
if optimization == "pretrained" and cnn_path and cnn_path.exists():
    state_dict = torch.load(cnn_path, map_location=DEVICE, weights_only=True)
    # 兼容旧版权重文件 (c1/c2/c3/c4/cls → block1/block2/block3/block4/classifier)
    _LEGACY_KEY_MAP = {
        "c1": "block1", "c2": "block2", "c3": "block3", "c4": "block4",
        "cls": "classifier",
    }
    new_state_dict = {}
    for k, v in state_dict.items():
        new_key = k
        for old_prefix, new_prefix in _LEGACY_KEY_MAP.items():
            if k.startswith(f"{old_prefix}."):
                new_key = f"{new_prefix}.{k[len(old_prefix)+1:]}"
                break
        new_state_dict[new_key] = v
    model.load_state_dict(new_state_dict)
```

### 优势
- 无需重新训练，节省 ~30 分钟 GPU 时间
- 向后兼容：新旧权重文件均可加载
- 修改范围极小（仅 1 处）

## BUG-002: 无监督预训练模式 labels_ 维度不匹配 [严重]

### 根因分析

`_run_unsupervised()` 第 984-987 行：

```python
if hasattr(model, "labels_"):        # ← KMeans/Agglomerative/Spectral 都有 labels_
    labels_pred = model.labels_      # ← 这是原始训练集(2000样本)的标签！
elif hasattr(model, "predict"):
    labels_pred = model.predict(X_scaled)
```

问题：
- `KMeans.labels_` = 训练时的 2000 个标签，当前数据只有 400 个样本 → 维度不匹配
- `Agglomerative/Spectral` 没有 `predict()` 方法，只能 `fit_predict()` → 必须回退到重新训练

### 修复方案: 分层处理策略

**文件**: `src/gradio_app.py` 第 980-991 行

**当前代码**:
```python
if optimization == "pretrained":
    model_path = UNSUP_DIR / f"{method}_best.joblib"
    if model_path.exists():
        model = joblib.load(model_path)
        if hasattr(model, "labels_"):
            labels_pred = model.labels_
        elif hasattr(model, "predict"):
            labels_pred = model.predict(X_scaled)
        status_msgs.append(f"✅ 已加载预训练模型: {model_path.name}")
    else:
        status_msgs.append(f"⚠️ 预训练模型未找到: {method}，现场训练")
        optimization = "manual"
```

**修改为**:
```python
if optimization == "pretrained":
    model_path = UNSUP_DIR / f"{method}_best.joblib"
    if model_path.exists():
        model = joblib.load(model_path)
        # 优先使用 predict()（对当前数据预测），而非 labels_（原始训练集标签）
        if hasattr(model, "predict"):
            try:
                labels_pred = model.predict(X_scaled)
                status_msgs.append(f"✅ 已加载预训练模型: {model_path.name}")
            except Exception as e:
                # 维度不匹配等异常 → 回退到 manual 模式
                status_msgs.append(
                    f"⚠️ 预训练模型 {model_path.name} 不兼容当前特征 "
                    f"({type(e).__name__})，回退到现场训练"
                )
                optimization = "manual"
                model = None
        else:
            # Agglomerative/Spectral 无 predict() 方法，必须重新训练
            status_msgs.append(
                f"⚠️ {method} 不支持 predict()，使用现场训练"
            )
            optimization = "manual"
            model = None
    else:
        status_msgs.append(f"⚠️ 预训练模型未找到: {method}，现场训练")
        optimization = "manual"
```

## BUG-004: 特征重要性图标签不可读 [轻微]

### 根因分析

`_plot_feature_importance()` 显示全部 108 个特征，纵轴标签严重重叠。

### 修复方案

**文件**: `src/gradio_app.py` `_plot_feature_importance()` 函数 (第 451-461 行)

**当前代码**:
```python
def _plot_feature_importance(importances, feature_names):
    if importances is None or len(importances) == 0:
        return None
    idx = np.argsort(importances)
    fig, ax = plt.subplots(figsize=(5.5, 4))
    ax.barh(range(len(idx)), importances[idx],
            color=plt.cm.Greens(np.linspace(0.3, 0.9, len(idx))))
    ax.set_yticks(range(len(idx)))
    ax.set_yticklabels([feature_names[i] for i in idx])
    ax.set_xlabel("重要性"); ax.set_title("特征重要性")
    plt.tight_layout()
    return fig
```

**修改为**:
```python
def _plot_feature_importance(importances, feature_names, top_n=20):
    """绘制特征重要性图，默认只显示 Top N 个特征。"""
    if importances is None or len(importances) == 0:
        return None
    # 只取 Top N
    n_show = min(top_n, len(importances))
    idx = np.argsort(importances)[-n_show:]  # 取最大的 n_show 个
    fig_height = max(3, n_show * 0.25)
    fig, ax = plt.subplots(figsize=(5.5, fig_height))
    ax.barh(range(n_show), importances[idx],
            color=plt.cm.Greens(np.linspace(0.3, 0.9, n_show)))
    ax.set_yticks(range(n_show))
    ax.set_yticklabels([feature_names[i] for i in idx])
    ax.set_xlabel("重要性")
    ax.set_title(f"特征重要性 (Top {n_show})")
    plt.tight_layout()
    return fig
```

调用处 (第 722-723 行) 无需改动，因为 `top_n` 参数有默认值。

## BUG-005: DBSCAN 默认参数全标记为噪声 [轻微]

### 根因分析

默认 `eps=0.5, min_samples=5` 在高维标准化空间中过于严格，所有样本被标记为噪声 (label=-1)。

### 修复方案: 添加全噪声检测与自动 eps 调整

**文件**: `src/gradio_app.py` 第 1003-1006 行（DBSCAN 模型构建后）

在 `labels_pred = model.fit_predict(X_scaled)` 之后添加检测逻辑：

```python
# DBSCAN: 检测并处理全噪声情况
if method == "dbscan" and len(set(labels_pred)) <= 1 and all(l == -1 for l in labels_pred):
    # 使用 k-distance 方法自动估计 eps
    from sklearn.neighbors import NearestNeighbors
    k = int(params.get("min_samples", 5))
    nn = NearestNeighbors(n_neighbors=k)
    nn.fit(X_scaled)
    distances, _ = nn.kneighbors(X_scaled)
    k_distances = np.sort(distances[:, -1])
    # 取 k-distance 的 75 分位数作为 eps
    auto_eps = float(np.percentile(k_distances, 75))
    model = DBSCAN(eps=auto_eps, min_samples=k)
    labels_pred = model.fit_predict(X_scaled)
    status_msgs.append(f"⚠️ 默认 eps 过小，自动调整为 eps={auto_eps:.3f}")
```

## 安全加固备注

1. **`torch.load()` 安全**: BUG-001 修复中已采用 `weights_only=True`，避免反序列化任意 Python 对象导致的代码执行风险。`.pth` 文件仅包含 tensor 数据，兼容此模式。

2. **`joblib.load()` 安全**: BUG-002 中加载的 `.joblib` 模型文件（KMeans/GMM/Agglomerative/Spectral）来自本项目自身训练产出，属于可信来源。joblib 反序列化 sklearn 模型无安全替代方案（sklearn 官方推荐使用 joblib），当前使用场景可接受。后续如需分发模型给第三方，建议考虑 ONNX 导出或 JSON 参数序列化方案。
